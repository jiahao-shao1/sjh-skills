# Agent Template Documentation

Two general-purpose agents generated by `init-skeleton.sh`.

## code-verifier

**Purpose**: Automatic quality check after code changes, before commit.

**Design decisions**:
- Model: haiku (fast, low cost — quality checks don't need strong reasoning)
- permissionMode: bypassPermissions (automated quality gate, no user approval needed)
- maxTurns: 15 (bounded execution to prevent runaway checks)
- Tools: Read/Grep/Glob/Bash only (no Edit/Write needed)
- Fixed check flow: identify changes → ruff lint/format → pytest → report
- Does not auto-fix test failures, only reports

**Customization suggestions**:
- For non-Python projects, replace ruff with the corresponding linter (e.g., eslint, clippy)
- If the project has custom test commands, modify the pytest section
- Can add project-specific exclude directories in the "identify changed files" step

## planner

**Purpose**: Codebase research assistant for brainstorming and writing-plans phases.

**Design decisions**:
- Model: opus (needs deep code understanding, strong reasoning)
- permissionMode: plan (read-only research, no code modifications needed)
- Tools: Read/Grep/Glob only (read-only, no code modifications)
- Outputs structured research findings (relevant files, existing patterns, suggestions, risks)
- Not a standalone planner — just a research tool

**Customization suggestions**:
- Add project-specific focus directories in the "systematic search" section
- If the project has domain expert agents, add routing logic in "notes"

## Template for Adding New Agents

```markdown
---
name: {agent-name}
description: {one-line description of purpose and trigger scenarios}
model: {haiku|sonnet|opus}
memory: {project|session}        # optional: project = cross-session, session = ephemeral
permissionMode: {bypassPermissions|plan|default}  # optional: controls autonomy level
maxTurns: {N}                    # optional: bound execution (recommended for automation agents)
tools:
  - Read
  - Grep
  - Glob
  # Add as needed: Bash, Edit, Write
---

# {Agent Name}

## When to Use

{specific usage scenario list}

## How It Works

### 1. {Step 1}
### 2. {Step 2}
### 3. Output Results

{output format template}

## Notes

- {key constraints}
- Chain hint: {if this agent's output feeds another agent, note it here — e.g., "After analysis, chain to prompt-proposer for fixes"}
```

### Frontmatter Field Guide

| Field | Values | When to Use |
|-------|--------|-------------|
| `model` | haiku / sonnet / opus | haiku=mechanical, sonnet=analysis, opus=deep reasoning |
| `memory` | project / session | `project` for cross-session diagnostics (RL experts, analyzers); `session` for ephemeral helpers |
| `permissionMode` | bypassPermissions / plan / default | `bypassPermissions`+`maxTurns` for automation; `plan` for read-only specialists |
| `maxTurns` | integer | Bound execution for automation agents (prevents runaway); typically 10-20 |

### Chain Hints

When agents form a sequential workflow, add a note in the upstream agent pointing to the downstream one:

```markdown
## Notes
- After analysis is complete, chain to `prompt-proposer` agent to generate improved versions
```

This helps the parent orchestrator route output correctly.
